## Memories

This optional section enables durable Memgraph Memory usage for **`{{PROJECT_NAME}}`**. Every query
MUST include `project: '{{PROJECT_NAME}}'`.

### Memory Triggers

- **NO DELEGATION:** never delegate memory state queries, lifecycle changes, or memory/code-ref lookups to subagents. Use Memgraph yourself.
- **Status/pending-work:** run Memory Orientation first, then Git if local changes matter. Use Git alone only when explicitly asked for Git-only status.
- **Orientation reuse:** reuse session-scoped orientation unless memory changed, the user asks for refresh, or scope changed.
- **RAG-first memory discovery:** use Memory RAG for broad history/context prompts: prior/similar work, task history, status/pending work, "when was this changed", or unfamiliar-subsystem context. For known Memory ids, exact status/type lists, lifecycle updates, task close, `CodeRef` follow-up, and scoped Orientation, use exact Memory queries.
- **Hypothesis-driven RAG:** generate 1-3 concise semantic queries from suspected decisions, task history, observed symbols/errors, risks, or verified clues. Each query must aim at one exact Memory/CodeRef lookup or source read. After two unhelpful searches for the same question, stop and use exact Orientation or text search. The limit resets only after exact verification changes the question or reveals a new concrete clue. Briefly report prompts that materially shaped the result.
- **Memory investigation budget:** do not run Memory RAG or Orientation for ordinary code work unless the user asks for prior history, status, pending work, existing Memory ids, or unfamiliar-subsystem context. Initial Memory RAG is index-only: return ids, titles, statuses, labels, and similarity, not `chunk.text` or full body fields. Start with at most 5 hits, then fetch exact records only for selected relevant IDs.
- **Task tracking:** for multi-step implementation, debugging, refactoring, documentation, dependency, test, or coverage work, create/update a `Task` as `doing` before edits, even if you expect to finish in the same response. Close any Task you touched as `done`, `blocked`, or `cancelled` before final response and verify it.
- **Lifecycle changes:** immediately update Task/Risk/Question/Decision/ADR/Idea status before proceeding.
- **Code-related memory:** when creating Task/Decision/Finding/Rule/ADR/Risk/Idea about code, create at least one `CodeRef` and link it through `(:MemoryNode)-[:REFERS_TO]->(:CodeRef)-[:RESOLVES_TO]->(:Code|:Package|:File|:Class|:Interface|:Annotation|:Method|:Field)`.
- **Session Memory embedding refresh:** every Memory node created or materially updated in the current session needs a refreshed `MemoryChunk`, stale embedding properties cleared when text changes, embedding generated by Memgraph, and metadata stamped in a separate statement.

### Memory RAG Vectors (only if RAG has embeddings)

Use `:MemoryChunk` only for broad project-knowledge discovery. RAG hits are not authority; fetch the
canonical Memory node and linked `CodeRef` targets before acting. If no compatible
`memory_chunk_embedding_v1` index exists or hits are irrelevant, fall back to scoped exact
Orientation and say why.

```cypher
SHOW VECTOR INDEX INFO;

CALL mg.procedures() YIELD name
WITH name
WHERE name CONTAINS 'embeddings' OR name CONTAINS 'vector_search'
RETURN name
ORDER BY name;

CALL embeddings.text(['<hypothesis-specific memory query>'], {}) YIELD embeddings
WITH embeddings[0] AS queryVector
CALL vector_search.search('memory_chunk_embedding_v1', 5, queryVector)
YIELD node AS chunk, similarity
WITH chunk, similarity
WHERE chunk.project = '{{PROJECT_NAME}}'
MATCH (memory {project: '{{PROJECT_NAME}}'})-[:HAS_RAG_CHUNK]->(chunk)
RETURN labels(memory) AS type, memory.id AS id, memory.title AS title,
       memory.status AS status, chunk.sourceLabel AS sourceLabel,
       chunk.sourceId AS sourceId, similarity
ORDER BY similarity DESC;
```

Follow relevant hits:

```cypher
MATCH (memory {project: '{{PROJECT_NAME}}', id: '<memory-id>'})
OPTIONAL MATCH (memory)-[:REFERS_TO]->(ref:CodeRef)-[:RESOLVES_TO]->(target)
RETURN labels(memory) AS type, memory.id AS id, memory.title AS title,
       memory.status AS status, ref.targetType AS targetType,
       ref.key AS key, labels(target) AS targetLabels;
```

### Orientation

When required and RAG is available, search `MemoryChunk` first, keep discovery index-only, then fetch
exact records and CodeRefs for selected IDs. When RAG is absent/irrelevant, use scoped exact queries:

```cypher
MATCH (m:Memory {project: '{{PROJECT_NAME}}'})-[:HAS_RULE]->(r:Rule)
RETURN r.id, r.severity, r.description ORDER BY r.severity;

MATCH (m:Memory {project: '{{PROJECT_NAME}}'})-[:HAS_FINDING]->(f:Finding)
WHERE f.status = 'open'
RETURN f.id, f.type, f.summary ORDER BY f.id;

MATCH (m:Memory {project: '{{PROJECT_NAME}}'})-[:HAS_TASK]->(t:Task)
WHERE t.status IN ['todo', 'doing', 'blocked']
RETURN t.id, t.title, t.status, t.priority ORDER BY t.priority, t.status, t.id;

MATCH (m:Memory {project: '{{PROJECT_NAME}}'})-[:HAS_QUESTION]->(q:Question)
WHERE q.status = 'open'
RETURN q.id, q.title ORDER BY q.id;

MATCH (m:Memory {project: '{{PROJECT_NAME}}'})-[:HAS_RISK]->(r:Risk)
WHERE r.status = 'open'
RETURN r.id, r.title, r.severity ORDER BY r.severity, r.id;
```

Context orientation is index-first. Fetch `content` only for relevant IDs/topics/keywords unless the
task is broad memory cleanup:

```cypher
MATCH (m:Memory {project: '{{PROJECT_NAME}}'})-[:HAS_CONTEXT]->(c:Context)
WHERE c.id IN ['CTX-<id>'] OR c.topic IN ['<topic>']
RETURN c.id, c.title, c.topic, c.content, c.source
ORDER BY c.id;
```

### Memory Schema

**Strict:** no extra properties. Use `datetime()` for `createdAt`/`updatedAt`, not
`localDateTime()`. Existing graphs may contain legacy timestamp strings; normalize before ordering:

```cypher
MATCH (t:Task {project: '{{PROJECT_NAME}}'})
WITH t, toString(coalesce(t.updatedAt, t.createdAt, '')) AS changedAt
RETURN t.id, t.title, changedAt
ORDER BY changedAt DESC
LIMIT 20;
```

Cypher drops variables not carried through `WITH`; after `CALL ... YIELD`, explicitly keep every
needed variable. Put required `MATCH` clauses before `OPTIONAL MATCH`; Memgraph rejects `MATCH`
after `OPTIONAL MATCH`.

Memory nodes and properties:

| Label | Key props | Additional properties |
| --- | --- | --- |
| `:Memory` | `project` | - |
| `:Decision` | `id`, `project` | `title`, `topic`, `status`, `rationale`, `consequences`, `createdAt`, `updatedAt` |
| `:ADR` | `id`, `project` | `number`, `title`, `status`, `context`, `decision`, `consequences`, `createdAt`, `updatedAt` |
| `:Rule` | `id`, `project` | `title`, `topic`, `severity`, `description`, `createdAt`, `updatedAt` |
| `:Context` | `id`, `project` | `title`, `topic`, `content`, `source`, `createdAt`, `updatedAt` |
| `:Finding` | `id`, `project` | `title`, `topic`, `type`, `status`, `summary`, `evidence`, `createdAt`, `updatedAt` |
| `:Task` | `id`, `project` | `title`, `status`, `priority`, `description`, `createdAt`, `updatedAt` |
| `:Risk` | `id`, `project` | `title`, `topic`, `severity`, `status`, `mitigation`, `createdAt`, `updatedAt` |
| `:Question` | `id`, `project` | `title`, `status`, `answer`, `createdAt`, `updatedAt` |
| `:Idea` | `id`, `project` | `title`, `topic`, `status`, `notes`, `createdAt`, `updatedAt` |
| `:CodeRef` | `project`, `targetType`, `key` | - |
| `:MemoryChunk` | `id`, `project` | `sourceLabel`, `sourceId`, `text`, `textHash`, `embedding`, `embeddingModel`, `embeddingDimensions`, `embeddingDirty`, `createdAt`, `updatedAt` |

Controlled values: Decision/ADR `status` = `proposed`, `accepted`, `rejected`, `superseded` (ADR also `draft`);
Rule `severity` = `hard`, `soft`, `recommendation`; Finding `type` = `bug`, `perf`,
`constraint`, `security`, `status` = `open`, `resolved`, `obsolete`; Task `status` = `todo`,
`doing`, `done`, `blocked`, `cancelled`, `priority` = `0` critical through `4` none; Risk
`severity` = `low`, `medium`, `high`, `critical`, `status` = `open`, `mitigated`, `accepted`,
`obsolete`; Question `status` = `open`, `answered`, `obsolete`; Idea `status` = `proposed`,
`accepted`, `rejected`, `obsolete`.

IDs use `DEC-`, `ADR-<n>-`, `RULE-`, `FIND-`, `TASK-`, `RISK-`, `CTX-`, `Q-`, `IDEA-` plus
`<topic>-<name>`.

```text
(:Project)-[:HAS_MEMORY]->(:Memory)-[:HAS_*]->(:Decision|:ADR|:Rule|:Context|:Finding|:Task|:Risk|:Question|:Idea)
(:Decision|...)-[:REFERS_TO]->(:CodeRef)-[:RESOLVES_TO]->(:Code|:Package|:File|:Class|:Interface|:Annotation|:Method|:Field)
(:Decision|...)-[:HAS_RAG_CHUNK]->(:MemoryChunk)
```

`CodeRef.key`: `Code` uses `java`, `js`, or `python`; `Package` uses `java:<package>`,
`js:<package>`, or `python:<package>`; `File` uses path; types/fields use FQN; `Method` uses
signature. `MemoryChunk` is derived RAG data, not authority; its `text` should summarize type,
title, topic/status/severity/priority, body fields, and CodeRefs.

### Memory Policy

Memory is not a changelog. Store only information useful for future decisions, investigations, or
implementation. A `Task` is for durable tracking, explicit follow-up, unfinished/blocked work, or
multi-step file edits; create it even when finished in the same response. Do not create a Task for
one-off read-only lookups or pure verification.

Use:
- `Decision` for design/implementation choices.
- `ADR` for architecture direction.
- `Rule` for future constraints.
- `Finding` for bugs, performance issues, wrong assumptions, or limitations.
- `Context` for durable subsystem behavior, operational caveats, recurring failure modes, or reusable investigation summaries; keep content concise and current.
- `Task` for tracked active work or unfinished follow-up.
- `Question` for open questions.
- `Risk` for new/discovered risks.

### Saving Memory

With interactive `mgconsole`, keep writes short; split lifecycle/content/link/verification into
separate statements.

Create/update the Memory node first:

```cypher
MERGE (m:Memory {project: '{{PROJECT_NAME}}'})
MERGE (d:Decision {id: 'DEC-<topic>-<name>', project: '{{PROJECT_NAME}}'})
SET d.title = '<title>', d.topic = '<topic>', d.status = 'accepted',
    d.rationale = '<rationale>', d.consequences = '<consequences>',
    d.createdAt = coalesce(d.createdAt, datetime()), d.updatedAt = datetime()
MERGE (m)-[:HAS_DECISION]->(d);
```

Link code in a separate query. `MATCH` existing code nodes; do not `MERGE` code nodes inline.

```cypher
MATCH (d:Decision {id: 'DEC-<topic>-<name>', project: '{{PROJECT_NAME}}'})
MATCH (c:Class {fqn: '<fqn>', project: '{{PROJECT_NAME}}'})
MERGE (ref:CodeRef {project: '{{PROJECT_NAME}}', targetType: 'Class', key: '<fqn>'})
MERGE (d)-[:REFERS_TO]->(ref)
MERGE (ref)-[:RESOLVES_TO]->(c);
```

When creating or materially updating a Memory node, also refresh one derived `MemoryChunk` for
semantic RAG discovery. Store exact chunk text and a stable hash, clear old embedding properties
when text changes, then create the embedding with Memgraph. Use the source Memory node as authority.
Compute `textHash` outside Cypher with SHA-256, or use a documented utility procedure such as
`util_module.md5(text)` when available; never call `sha256()` in Memgraph. When using Node.js
snippets, avoid top-level `createHash` declarations; keep imports/helpers scoped so repeated REPL
calls do not redeclare persistent bindings.

```cypher
MATCH (d:Decision {id: 'DEC-<topic>-<name>', project: '{{PROJECT_NAME}}'})
MERGE (chunk:MemoryChunk {id: 'MCH-DEC-<topic>-<name>', project: '{{PROJECT_NAME}}'})
SET chunk.sourceLabel = 'Decision',
    chunk.sourceId = d.id,
    chunk.text = '<type/title/topic/status/body/code-ref-summary text>',
    chunk.textHash = $textHash,
    chunk.createdAt = coalesce(chunk.createdAt, datetime()),
    chunk.updatedAt = datetime(),
    chunk.embeddingDirty = true
REMOVE chunk.embedding, chunk.embeddingModel, chunk.embeddingDimensions
MERGE (d)-[:HAS_RAG_CHUNK]->(chunk);
```

Ensure `memory_chunk_embedding_v1` exists before embedding refresh; create it only when absent:

```cypher
CREATE VECTOR INDEX memory_chunk_embedding_v1 ON :MemoryChunk(embedding)
WITH CONFIG $config;
```

Session Memory embedding refresh: set `$ids` to chunks created/updated in this session. This
statement intentionally ends with `RETURN` immediately after `embeddings.node_sentence()`:

```cypher
MATCH (chunk:MemoryChunk {project: '{{PROJECT_NAME}}'})
WHERE chunk.id IN $ids
  AND chunk.text IS NOT NULL
  AND (chunk.embedding IS NULL
    OR chunk.embeddingModel IS NULL
    OR chunk.embeddingModel <> $modelName
    OR chunk.embeddingDimensions IS NULL
    OR chunk.embeddingDimensions <> $dimension)
WITH chunk
ORDER BY chunk.id
WITH collect(chunk) AS chunks
WITH chunks, [chunk IN chunks | chunk.id] AS embeddedIds
CALL embeddings.node_sentence(chunks, $config)
YIELD success, dimension
RETURN success AS success, dimension AS dimension, embeddedIds AS ids;
```

Stamp metadata in a separate statement:

```cypher
MATCH (chunk:MemoryChunk {project: '{{PROJECT_NAME}}'})
WHERE chunk.id IN $ids

    chunk.embeddingDimensions = $dimension,
    chunk.embeddingDirty = false,

```
